Introduction

  • A Hash Table in different Programming languages also called: Hashes, maps, hash maps, dictionaries and associative arrays
  • Hash tables provide us with fast reading, in other words instant lookup with O(1)O(1)O(1)﻿ time complexity.

  • Example: 
menu = {
				"French Fries" : 2.99,
				"Sandwich" : 6.99,
				"Drink": 2.19
			}
  • A hash table is a list of paired values. The first item in each pair is called key, and the second item is called value. So, its {key, value} pair.
  • In the code above, “French Fries” can be one of the key and the value associated with that key is 2.99

Hashing with Hash Functions

  • A key is in the form of string, and the function takes each character of string and converts it to the numbers, this process of conversion is done by Hash Functions.

  • There are many different type of hash functions, most common is Remainder Hashing.
  • No functions can be used that use random numbers or the current time as a part of calculation.

Dealing with Collisions

  • Trying to add data to a cell that is already filled is knows as a collision.
  • One way to handle this issue is that to use Separate chaining.

    Separate Chaining

    • When a collision occurs, instead of placing a single value in the cell, it places in it a reference to an array

    The array contains key and value associated with it, and when we hash the key to retrieve the value, the algorithm will search through the array linearly to find the value.

    In this case of collision, our code turns out to be O(N)O(N)O(N)﻿ in the worst-case performance. And because of this it is critical that a hash table is designed that way it will have few collisions and therefore typically perform lookups in O(1)O(1)O(1)﻿ rather than O(N)O(N)O(N)﻿ time.

Making an Efficient Hash Table

  1. How much data we’re storing
  1. How many cells are available
  1. Which hash function we’re using

  • If you have a lot of data and only a few cells, there will be many collisions and hash table will lose its efficiency.

  • If the hash function is giving same values continuously then all of the data will be stored as chained and will lose its efficiency.

The Great Balancing Act

  • Efficiency goes up as its number of collisions goes down.
  • Best way to avoid collisions would be to have a hash table with a large number of cells.
  • A good hash table strikes a balance of avoiding collisions while not consuming lots of memory.
  • To accomplish this, there is a rule of thumb that for every 7 elements stored in a table it should have 10 cells. If we are storing 14 elements we would have 20 cells.
    • This ratio of data to cell is called the load factor, and the ideal load factor is 0.7 (7 elements / 10 cells).
